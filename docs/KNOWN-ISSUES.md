## Активные проблемы

### 1. Mistral — 401 Unauthorized

**Компонент:** `server/engines/mistral.ts`  
**Серьёзность:** средняя — 1 из 8 движков недоступен  
**Статус:** требуется действие

**Описание:**  
Движок Mistral Small возвращает HTTP 401 при каждом запросе. Текущий API-ключ `MISTRAL_API_KEY` истёк или был деактивирован на стороне Mistral AI.

Движок присутствует в `allEngines` и фильтруется при запуске только если ключ полностью отсутствует — при установленном, но невалидном ключе движок попытается отправить запрос и вернёт ошибку.

**Влияние:**  
- В ответе `POST /api/translate` движок `mistral` появится с `error: "401 Unauthorized"` или аналогичным
- Ensemble всё равно работает на оставшихся 7 движках
- Selftest для Mistral будет показывать `available: false, successRate: 0`

**Решение:**  
1. Зайти на [console.mistral.ai](https://console.mistral.ai/) и создать новый API-ключ
2. Обновить переменную окружения `MISTRAL_API_KEY` в `.env` на VPS
3. Перезапустить контейнер: `docker compose restart`

### 2. Gemini Critic — возможное превышение таймаута

**Компонент:** `server/engines/postprocess.ts` → `getCritique()`  
**Серьёзность:** низкая — не влияет на результат  
**Статус:** допустимо (non-fatal)

**Описание:**  
Gemini 2.5 Flash используется как независимый критик с таймаутом 18 секунд (`CRITIC_TIMEOUT_MS = 18000`). При высокой нагрузке на Gemini API или при длинных текстах критик может не уложиться в таймаут.

**Влияние:**  
- Критик возвращает `null`
- Ensemble GPT-4o продолжает работу без рецензии
- В SSE-событии `critic` поле `critique` будет отсутствовать
- Качество финального перевода незначительно снижается

**Рекомендации:**  
- Таймаут можно увеличить в `CRITIC_TIMEOUT_MS`, но это увеличит общее время ответа
- Текущее значение 18s — разумный компромисс между качеством и скоростью
- В логах сервера появится запись при truncation: `[critic] Gemini truncated (MAX_TOKENS)`

### 3. TTS (kk-KZ голос) — работает только в Chrome/Edge

**Компонент:** `client/src/pages/translate-v2.tsx` → TTS Player  
**Серьёзность:** низкая — базовый TTS работает везде  
**Статус:** ограничение браузерного API

**Описание:**  
Компонент TTS Player использует `window.speechSynthesis` с голосом `kk-KZ` (казахский язык). Голос `kk-KZ` доступен только в браузерах на движке Chromium: Chrome 88+, Edge 88+, Opera.

В других браузерах:
- **Firefox** — нет голоса `kk-KZ`; `speechSynthesis.getVoices()` не вернёт казахский голос
- **Safari** — нет поддержки казахского голоса на большинстве платформ
- **Mobile browsers (Android Chrome)** — голос `kk-KZ` обычно доступен
- **iOS Safari** — голос `kk-KZ` недоступен

**Поведение при недоступности голоса:**  
TTS Player либо молчит, либо использует fallback-голос (обычно системный дефолт), который будет читать казахский текст с неправильным произношением.

**Рекомендации:**  
- Добавить проверку наличия голоса и показывать уведомление пользователю при его отсутствии
- Долгосрочно: интегрировать серверный TTS (напр. Yandex SpeechKit с поддержкой KK)

### 4. Self-test — API-вызовы ко всем движкам

**Компонент:** `server/selftest.ts`  
**Серьёзность:** низкая — только в production  
**Статус:** ожидаемое поведение

**Описание:**  
Self-test запускает полный конвейер перевода для каждого из 10 бенчмарк-кейсов каждые 6 часов в production. Каждый прогон вызывает все настроенные движки (включая платные API) и запускает GPT-4o для Ensemble + MQM самооценки.

**Оценочные расходы за 1 прогон:**
- ~10 запросов к каждому из 8 движков
- ~10 запросов к GPT-4o (Ensemble, один на тест)
- ~10 запросов к GPT-4o (Self-eval, один на тест)
- ~10 запросов к Gemini 2.5 Flash (Critic)

При активных 8 движках + GPT-4o Ensemble + GPT-4o Self-eval + Gemini Critic: ~100+ API-запросов за прогон.

**Рекомендации:**  
- Уменьшить интервал или количество тест-кейсов, если расходы API критичны
- `startPeriodicTesting(6)` — параметр можно изменить в `routes.ts`
- В development режиме selftest не запускается (защита через `NODE_ENV === "production"`)
- Для ручного тестирования использовать `POST /api/admin/selftest/run`

## Ограничения дизайна (не баги)

### 5. MQM-оценка не является reference-based

**Описание:**  
MQM Self-evaluation использует GPT-4o без эталонного перевода (reference-free). Оценка субъективна и зависит от качества модели-судьи. Это соответствует подходу WMT Shared Task с LLM-based evaluation, но не является строгой лингвистической метрикой.

**Следствие:** Оценки разных прогонов для одного текста могут незначительно различаться (±1 балл).

---

### 6. Нет аутентификации для Admin API

**Описание:**  
Эндпоинты `/api/admin/*` не защищены паролем или токенами в текущей версии. Доступ к ним не ограничен на уровне приложения.

**Рекомендация:**  
Защитить через Nginx (Basic Auth или IP allowlist):
```nginx
location /api/admin {
    allow 1.2.3.4;  # ваш IP
    deny all;
    proxy_pass http://localhost:5000;
}
```

---

### 7. allResults сохраняется в БД как JSON-строка

**Описание:**  
Поле `allResults` в таблице `translations` хранится как `TEXT` (JSON-строка), а не как отдельные строки. Это упрощает схему, но затрудняет SQL-запросы по результатам отдельных движков.

**Следствие:** Для анализа результатов по движкам нужно парсить JSON в приложении.

## Устранённые проблемы (исправлено в v4.0.0)

| Проблема | Исправление |
|----------|-------------|
| `selectBest()` крашился при пустом массиве | Возвращает `null`; маршруты возвращают 503 |
| DeepL в списке движков (не поддерживает KK) | Удалён из `allEngines` |
| Self-test вызывал все движки дважды | Один `translateWithAll()` на тест |

| SSE: событие `ensemble` отправлялось до завершения критика | Правильная последовательность событий |
| Selftest засчитывал тест пройденным при score=null | Требуется score >= 7 и score !== null |
| mdash в элементах диапазонов UI | Заменён на ndash |
